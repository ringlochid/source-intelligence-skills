# Router Cheat Sheet

Use this only when `routing-matrix.md` is too terse.

## Source plan rules

- Start with one lane when possible.
- Add a second lane only for verification or a distinct signal class.
- Do not use raw source-count targets; stop when strong evidence is sufficient.
- Use official/primary sources before commentary.
- Use social/community sources as leads or sentiment only.
- Use `source-briefing` for synthesis after gathering.
- Use `source-visual` only when visual evidence/output matters.
- Use `source-artifact` only when files are requested.

## Common pairings

- breaking + market: market-moving news
- tech + github: developer tooling/repo momentum
- research + github: paper implementation/adoption
- product + tech/github: product opportunity scans
- product + social-signal: weak sentiment only

## Route cards

- "topic digest/RSS" → `source-topic`, then briefing.
- "latest regulation/news" → `source-breaking`, then briefing.
- "AI/dev ecosystem" → `source-tech`, then briefing.
- "paper + implementations" → `source-research` + optional `source-github`.
- "repo/release health" → `source-github`, then briefing.
- "competitor/product scan" → `source-product` + optional `source-tech`.
- "BTC/ticker/earnings" → `source-market`, verify data separately.
- "X/Reddit buzz" → `source-social-signal`, signal not proof.
- "recurring curated sources" → `source-watchlist`, then collector/briefing.
- "make a report/file" → gather + briefing first, then `source-artifact`.
- "chart/screenshot/figure evidence" → collector first, then `source-visual`.

## Stop cues

- quick: enough strong sources answer the question
- standard: main source classes covered and claims cross-checked
- deep: contradictions/counter-evidence checked
- watchlist: new/material changes reviewed
